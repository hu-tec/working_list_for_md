# HutechC 사내 Studio 홈페이지 분석

## 1. 분석 개요

| 항목 | 내용 |
| --- | --- |
| 대상 | HutechC 사내 Studio |
| 주소 | `http://54.116.15.136:81` |
| 확인일 | 2026-05-26 |
| 분석 범위 | 비로그인 초기 화면과 공개 응답 상태, 사용자가 직접 로그인한 세션의 내부 화면 구조(읽기 전용) |
| 확인 경로 | `/`, `/dashboard.html`, `/admin.html`, `/app/`, `/api/auth/me`, `/api/interviews`, `/app/work-log`, `/app/work-hub`, `/app/company-guidelines`, `/app/recruitment`, `/app/rules`, `/app/manual-list`, `/app/admin-system`, `/app/attendance` |
| 캡처 | [hutech-home.png](../../.captures/hutech-home.png) |

공개 범위 확인 후, 사용자가 분석 전용 브라우저에서 직접 로그인한 세션에 연결하여 내부 화면을 읽기 전용으로 확인했다. 데이터 등록·수정·삭제는 실행하지 않았으며, 업무 항목·직원명·지원자 정보 등 개별 데이터 값은 수집하거나 문서화하지 않았다. 하나의 로그인 세션만 확인했으므로 역할별 권한 차이와 실제 데이터 품질은 평가 범위에서 제외했다.

## 2. 한줄 진단

문서 자동화 서비스의 가치를 설득하는 랜딩과 내부 업무 도구가 한 서버에 혼재한다. 화면 완성도는 양호하지만, 사내 시스템이라면 가장 먼저 처리해야 할 문제는 공개된 대시보드·관리 경로와 API 접근 통제이다.

## 3. 관찰된 화면과 정보 구조

### 초기 화면

초기 화면은 `Premium AI Platform` 브랜드로 표현되며, 사내 Studio 명칭은 `/app/` 애플리케이션 타이틀에서 확인된다. 랜딩은 다음 순서로 구성된다.

| 영역 | 관찰 내용 | 역할 |
| --- | --- | --- |
| 헤더 | 서비스, 전문가 네트워크, 내부 운영팀, 로그인, AI Studio, 대시보드 바로가기 | 핵심 메뉴와 내부 도구 진입 |
| 히어로 | "사내 모든 문서를 하나의 AI 워크스페이스로", 테스트 CTA, 소개서 다운로드 | 가치 제안과 전환 유도 |
| 제품 시연 패널 | 카테고리 선택, 생성된 문서 초안, 전체 화면 편집하기 | 기능 이해 보조 |
| 지표 | 도입 기업 120+곳, 처리 시간 `-68%`, CS 응답 `-55%` | 신뢰 확보 |
| 서비스 | 정책·규정 생성, 계약서 템플릿, CS 스크립트 추천 | 기능 소개 |
| 전문가 연동 | 법률 자문 워크플로우, 규정 변경 모니터링 | 확장 서비스 |
| 내부 운영팀 | 역할별 접근 권한, 버전 관리, 승인·알림 | 사내 협업 시나리오 |

### 공개 연결 화면

| 경로 | 공개 상태 | 화면/기능 단서 |
| --- | --- | --- |
| `/dashboard.html` | `200 OK` | `시온 프로세스 대시보드`, AI Studio, 데이터 관리, 면접 안내, 메뉴얼 관리 링크; 직군별 면접/업무 단계 및 메뉴얼 구조 |
| `/admin.html` | `200 OK` | 관리 화면 HTML이 비로그인 응답 |
| `/app/` | `200 OK` | React 앱 엔트리, 타이틀 `HutechC 사내 Studio` |
| 앱 번들 | 공개 | 업무일지, 사내업무지침, 채용관리, 출퇴근 관리, 미팅 관리, 규정관리, 면접대시보드 등의 라우트/문구 |

### 추정 사용자 흐름

```text
랜딩
  -> AI Studio (/app/) -> 로그인 기반 사내 앱
  -> 대시보드 (/dashboard.html) -> 데이터 관리 / 면접 안내 / 메뉴얼 관리
  -> 페이지 내 소개 섹션 탐색
```

`/app/` 번들에는 `/api/auth/me`, `/api/auth/login`, `/api/auth/logout` 호출이 존재하므로 앱 영역은 인증 흐름을 고려해 개발된 것으로 판단된다. 반면 정적 대시보드와 관리 경로는 별도 방어 없이 공개 응답한다.

### 로그인 후 사용자 흐름

```text
로그인
  -> 업무일지에서 오늘 업무 기록/확인
  -> 업무 총괄에서 부서별 진행 현황 점검
  -> 사내업무지침 / 규정 / 매뉴얼에서 기준 문서 조회·관리
  -> 채용관리 / 출퇴근 관리에서 인사 업무 처리
  -> 관리자 통합에서 플랫폼·사이트·권한 설정
```

로그인 세션에서 `/api/auth/me`는 인증된 사용자 정보를 반환했다. 값은 수집하지 않았으나 사용자 표시 정보와 `tier`, `status` 성격의 속성이 존재하고 명시적인 `role` 필드는 확인되지 않았다. 고권한 기능이 많은 만큼 권한 판단을 화면 표시가 아니라 서버 측 세부 권한으로 검증할 필요가 있다.

## 6. 기능 및 운영 평가

랜딩에서 제시하는 제품 범위는 정책·계약·CS 문서 생성, 전문가 검토, 승인·이력 관리까지 넓다. 로그인 후에는 채용, 출퇴근, 업무일지, 규정, 매뉴얼, 관리자 시스템까지 실제 화면 구조로 확인된다. 즉 현재 사이트는 단일 문서 AI 도구보다 사내 운영 포털이자 일부 플랫폼 관리 콘솔에 가까운 범위를 가진다.

이 범위를 유지한다면 홈의 메시지도 다음 두 사용자군을 구분해야 한다.

| 사용자군 | 필요한 첫 행동 | 제안 |
| --- | --- | --- |
| 일반 직원/관리자 | 맡은 업무와 승인 항목으로 진입 | 로그인 후 개인 대시보드를 기본 진입점으로 제공 |
| 서비스 소개 방문자 | 제품 이해와 문의 | 별도 공개 랜딩에서 데모·문의·소개 자료 제공 |
| 시스템 관리자 | 권한·워크플로우·사이트 설정 | 별도 관리자 진입점, 추가 인증, 변경 감사 제공 |

## 7. 보안 및 기술 관찰

### 즉시 검토가 필요한 사항

| 우선순위 | 관찰 사실 | 위험 |
| --- | --- | --- |
| Critical | 인증 없이 `/api/interviews`가 `200 OK`, `application/json`, 본문 길이 `1789`로 응답 | 면접 관련 데이터가 노출될 가능성이 높음. 개인정보 또는 평가 정보라면 즉시 사고 대응 대상 |
| High | 인증 없이 `/admin.html`, `/dashboard.html`이 `200 OK`로 응답 | 내부 메뉴·업무 구조 노출 및 관리 기능 접근 가능성 |
| High | 응답에 `Access-Control-Allow-Origin: *`가 표시됨 | 인증/데이터 API와 결합될 경우 외부 오리진에서 데이터 접근 범위 확대 가능 |
| Medium | 서비스가 공인 도메인 HTTPS가 아닌 IP 기반 HTTP로 제공됨 | 로그인 또는 업무 데이터가 오간다면 전송구간 보호와 신뢰성 부족 |
| High | 인증 후 화면에 채용·출퇴근·거래처·관리자 통합 기능이 함께 존재함 | 개인정보·경영정보·플랫폼 설정이 한 앱에 집약되어 권한 오설정의 영향이 큼 |

`/api/auth/me`는 본문 길이 4의 `200 OK`를 반환했고, 이는 비로그인 상태 표현일 수 있다. 그러나 `/api/interviews`처럼 데이터성 엔드포인트가 별도로 본문을 반환하는 점은 인증 적용 범위를 엔드포인트별로 전수 점검해야 함을 의미한다.

### 권고 조치

1. `/api/interviews`를 포함한 전체 `/api/*`에 서버 측 인증·권한 미들웨어를 적용하고 접근 로그를 확인한다.
2. 면접·인사·출퇴근·업무일지 데이터가 공개 응답된 적이 있는지 로그와 실제 데이터 종류를 점검하고, 필요 시 노출 대응 절차를 수행한다.
3. 공개 랜딩과 사내 운영 앱을 별도 호스트 또는 최소 별도 인증 게이트로 분리한다.
4. CORS를 허용 오리진 목록으로 축소하고 쿠키/세션 설정(`Secure`, `HttpOnly`, `SameSite`)을 검토한다.
5. HTTPS 도메인으로만 사내 앱을 제공하고 HTTP 접근은 리다이렉트한다.
6. 채용·출퇴근·거래처·관리자 시스템은 기능 단위 권한과 조회/내보내기/변경 감사 로그를 적용한다.
7. 사이트 생성, 권한/역할, 플러그인, 가격 정책 등 관리자 통합 기능에는 추가 인증 또는 관리자 전용 네트워크 정책을 검토한다.
