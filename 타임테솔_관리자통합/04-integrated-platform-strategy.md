# 3개 업무 시스템 통합 방안

## 1. 문서 목적

이 문서는 다음 3개 분석 문서를 바탕으로, 각 시스템의 역할을 하나의 통합 업무 플랫폼으로 재구성하는 방향을 설명한다.

| 문서 | 대상 | 핵심 성격 |
| --- | --- | --- |
| [01-hutechc-internal-studio-analysis.md](./01-hutechc-internal-studio-analysis.md) | HutechC 사내 Studio | 현대적 사내 업무 포털, AI 문서/규정/업무지침, 채용·근태·관리 기능 |
| [02-timesmedia-intranet-analysis.md](./02-timesmedia-intranet-analysis.md) | 타임스미디어 인트라넷 | 교육기관 운영 ERP, 학생·회계·근태·결재·공지·온라인미팅·출강·홈페이지 관리 |
| [03-work-studio-analysis.md](./03-work-studio-analysis.md) | Work Studio | 여러 홈페이지의 접수·신청서·콘텐츠·프로젝트 현황을 관리하는 운영 허브 |

통합의 목표는 기존 기능을 단순히 한 화면에 모으는 것이 아니다. 중복 기능을 정리하고, 핵심 업무 데이터 모델을 통일하며, 역할별 권한과 보안을 먼저 세운 뒤, 사용자가 매일 처리해야 하는 업무를 하나의 흐름으로 연결하는 것이다.

## 2. 통합 결론

3개 시스템은 다음처럼 역할을 나누어 통합하는 것이 적합하다.

```text
통합 업무 플랫폼
  -> 공개 홈페이지/신청 채널
  -> Work Studio: 접수·문의·신청서·콘텐츠 운영
  -> Times 운영 코어: 학생·강좌·회계·수료·출강·확인서
  -> HutechC Studio: 업무일지·규정·매뉴얼·AI 문서·채용·근태·관리자 콘솔
  -> 공통 기반: 인증, 권한, 감사로그, 파일, 알림, 검색, 통계
```

가장 중요한 판단은 `타임스미디어 인트라넷`을 업무 데이터의 기준 시스템으로 삼는 것이다. 학생, 강좌, 납부, 수료, 출강, 확인서, 결재 등 교육기관 운영의 실제 원장이 이 시스템에 가장 많이 드러난다. 반면 `HutechC 사내 Studio`는 이 업무를 현대적인 UI와 AI 문서/규정/매뉴얼 체계로 감싸는 상위 포털 역할이 적합하다. `Work Studio`는 외부 홈페이지에서 들어오는 신청·문의·콘텐츠 관리를 담당하는 프론트오피스 운영 계층으로 분리하는 편이 맞다.

| 시스템 | 현재 강점 | 통합 후 역할 | 통합 시 주의점 |

## 5. 핵심 데이터 모델 통합

통합의 난이도는 UI보다 데이터 모델에 있다. 우선 다음 기준 엔티티를 정해야 한다.

| 기준 엔티티 | 설명 | 우선순위 |
| --- | --- | --- |
| 사용자/User | 직원, 강사, 관리자, 회계, 센터 담당자, 마케팅 담당자 | P0 |
| 조직/센터/부서 | 본부, 센터, 팀, 직무, 역할 | P0 |
| 학생/문의자 | 홈페이지 신청자, 상담 대상자, 수강생, 졸업생 | P0 |
| 강좌/기수 | 과정, 세부강좌, 센터, 개강일, 졸업일, 정원 | P0 |
| 신청/문의 | 외부 홈페이지에서 들어온 폼 데이터, 견적, 상담 요청 | P1 |
| 납부/회계 | 입학금, 수업료, 환불, 현금영수증, 비일치 상태 | P1 |
| 업무/프로젝트 | 업무일지, 업무지시, 온라인미팅, 댓글, 마감, 담당자 | P1 |
| 문서/결재 | 기안서, 결재라인, 문서 양식, 지출결의, 보관 문서 | P1 |
| 규정/매뉴얼/서식 | 지침, 규정, 매뉴얼, 서식, 디자인물, AI 생성 문서 | P2 |
| 파일/첨부 | 신청서 첨부, 미팅 파일, 디자인물, 결재 증빙 | P2 |

기존 시스템의 데이터를 즉시 하나의 DB로 합치는 방식은 위험하다. 먼저 기준 ID와 상태값을 정의하고, 시스템 간 동기화 또는 읽기 전용 연동을 거친 뒤 단계적으로 마이그레이션해야 한다.

## 6. 권한 통합 원칙

3개 문서에서 공통적으로 가장 큰 위험은 인증·인가·민감 데이터 노출이다. 통합은 기능 개발보다 권한 모델을 먼저 설계해야 한다.

| 역할 | 기본 접근 범위 |
| --- | --- |
| 최고관리자 | 시스템 설정, 권한, 감사로그, 전체 데이터 접근 |
| 본부관리자 | 전체 학생/회계/결재/공지/프로젝트 조회와 승인 |
| 센터관리자 | 소속 센터 학생, 상담, 납부, 일정, 공지, 미팅 |
| 회계담당자 | 납부, 환불, 지출결의, 증빙, 비일치 검증 |
| 강사/출강담당 | 강의 일정, 레슨플랜, 출강 정보, 제한된 학생 정보 |
| 마케팅/홈페이지담당 | 신청서, 팝업, 콘텐츠, 접속통계, 디자인물 |
| 일반직원 | 본인 업무일지, 공지, 미팅, 매뉴얼, 제한된 검색 |
| 채용담당자 | 지원자, 면접, 평가, 채용 관련 문서 |

권한은 메뉴 노출만으로 처리하면 안 된다. 모든 API와 다운로드, 내보내기, 삭제, 상태 변경, 결재 처리에 서버 측 권한 검사가 필요하다.

## 8. 통합 후 대표 사용자 흐름

### 홈페이지 신청에서 수강생 등록까지

```text
외부 홈페이지 신청
  -> Work 계층에서 접수 생성
  -> 담당 센터/직원 배정
  -> 학생관리 후보로 전환
  -> 상담/예약 상태 기록
  -> 강좌/기수 배정
  -> 입학금/수업료 납부 확인
  -> 수강생 확정
```

### 일상 운영 흐름

```text
직원 로그인
  -> 오늘의 업무에서 미처리 접수, 결재 대기, 마감 미팅 확인
  -> 업무일지 작성
  -> 온라인미팅 댓글/완료 처리
  -> 필요한 규정/매뉴얼/서식 조회
  -> 결재 또는 보고서 작성
```

### 수료/확인서 발급 흐름

```text
강좌 종료
  -> 졸업/수료 상태 확인
  -> 납부/미납/환불 예외 검증
  -> Certi 또는 확인서 발급 요청
  -> 발급 상태와 결제 상태 기록
  -> 문서 보관
```

| Search | 메뉴, 학생, 신청, 문서, 미팅 통합 검색 |
| Legacy Adapter | 기존 Times/Work/HutechC 데이터 연동 및 단계적 이전 |

초기에는 `Legacy Adapter`를 두고 기존 시스템을 그대로 읽어오는 편이 안전하다. 기능별 검증이 끝난 뒤에 도메인 API와 데이터베이스를 분리 또는 통합한다.

## 10. 리스크와 대응

| 리스크 | 대응 |
| --- | --- |
| 통합 범위가 너무 커져 프로젝트가 장기화됨 | P0/P1/P2를 별도 마일스톤으로 나누고, 읽기 전용 통합부터 시작 |
| 기존 직원이 익숙한 레거시 화면을 잃음 | 초기에는 기존 화면 링크와 신규 화면을 병행하고, 사용량 기준으로 이전 |
| 학생/회계 데이터 정합성이 깨짐 | 데이터 소유권을 먼저 정하고, 쓰기 기능은 기준 시스템 하나로 제한 |
| 권한 설계가 느슨해짐 | 메뉴 권한이 아니라 API·필드·액션 권한으로 설계 |
| 파일/첨부가 보안 취약점이 됨 | 파일 서비스 분리, 확장자 허용목록, 악성 파일 검사, 실행 방지 적용 |
| 관리자 기능이 과다 노출됨 | 관리자 콘솔 분리, 추가 인증, IP 제한 또는 네트워크 제한 검토 |

## 11. 우선 착수 과제

1. `Work Studio`의 공개 기본 계정과 비인증 API를 즉시 차단한다.
2. `HutechC 사내 Studio`의 공개 대시보드/API와 관리자 경로 접근 통제를 점검한다.
3. `타임스미디어 인트라넷`의 PHP/세션/파일 업로드/권한 구조를 점검한다.
4. 통합 사용자·조직·역할·권한 매트릭스를 작성한다.
5. 학생/문의자/강좌/납부/업무/문서의 기준 데이터 모델을 정의한다.
6. 첫 통합 화면은 `오늘의 업무`로 제한해, 미처리 접수·결재 대기·미팅 마감·업무일지·일정을 한 화면에 모은다.
7. 이후 접수/문의 관리와 학생 조회를 연결하는 최소 기능부터 구현한다.

## 12. 최종 방향

최종 통합 플랫폼은 `업무를 처리하는 곳`, `규정과 문서를 찾는 곳`, `외부 신청을 내부 업무로 전환하는 곳`이 하나로 연결된 구조여야 한다. 단, 모든 기능을 한 메뉴에 쌓는 방식은 피해야 한다. 사용자는 역할에 맞는 오늘의 업무에서 시작하고, 시스템은 뒤에서 학생·접수·회계·결재·문서·미팅 데이터를 연결해야 한다.

따라서 통합 전략은 다음 순서가 가장 현실적이다.

```text
보안 차단
  -> 권한/용어/데이터 모델 정리
  -> 역할별 통합 홈
  -> 접수와 학생관리 연결
  -> 업무/결재/회계/문서 단계 이전
  -> 레거시 시스템 읽기 전용화 또는 폐기
```

이 순서로 진행하면 기존 업무를 중단하지 않으면서도, 세 시스템의 장점을 하나의 운영 플랫폼으로 흡수할 수 있다.
