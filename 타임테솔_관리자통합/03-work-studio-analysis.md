# Work Studio 홈페이지 분석

## 1. 분석 개요

| 항목 | 내용 |
| --- | --- |
| 대상 | Work Studio |
| 주소 | `http://54.116.15.136/` |
| 확인일 | 2026-05-26 |
| 분석 범위 | 공개 로그인 화면, 단일 페이지 HTML에 포함된 UI/기능 정의, API 응답 상태 |
| 확인 경로 | `/`, `/api/form_templates` |
| 캡처 | [work-login.png](../../.captures/work-login.png) |

사이트는 로그인 패널 안에 테스트용으로 보이는 계정 정보를 직접 표시하고 있다. 해당 계정을 사용해 데이터를 열람하거나 변경하지 않았으며, 공개 HTML과 API의 본문 미열람 응답 상태만으로 분석했다.

## 2. 한줄 진단

여러 홈페이지의 접수·콘텐츠·프로젝트 현황을 한곳에서 운영하려는 관리 허브지만, 화면에 표시된 기본 관리자 계정과 비인증 API 응답은 현재 기능 개선보다 앞서 차단해야 하는 중대한 위험이다.

## 3. 관찰된 사용자 경험

### 로그인 화면

검정 배경 중앙의 흰 카드에 `Work Studio`, `홈페이지 업무 관리`, 아이디/비밀번호 입력, 파란 로그인 버튼이 표시된다. 레이아웃은 단순하고 가독성이 좋지만 카드 하단에 `admin / admin`이 그대로 노출된다.

### HTML에 포함된 애플리케이션 구조

로그인 오버레이 뒤에 있는 UI와 스크립트에서 다음 메뉴·업무 구조가 확인된다. 이 내용은 공개된 프론트엔드 코드에 의한 기능 단서이며 권한별 실제 노출 여부는 별도 검증이 필요하다.

| 메뉴 | 확인된 목적/기능 |
| --- | --- |
| 실행 | 오늘 미처리, 단계별/역할별/상태별 집계, 신청 현황 행동 중심 홈 |
| 리스트 | 홈페이지별 신청/문의/견적 데이터 검색, 기간·상태 필터, 추가·수정·삭제, 메모 |
| 신청서 | 접수 현황, 템플릿 편집, 역할별·단계별 신청서 모듈 관리 |
| 분석 | 분포, 시계열, 피벗 등 분석 뷰 스타일/구현 단서 |
| 콘텐츠 | 문제, 커리큘럼, 교재 추가·수정·삭제 |
| 전체보기 | `p2-overview.html` 링크 |
| AnalyzeAI | 외부 AI 통합 분석 플랫폼 링크 |
| 홈생성 | `homepage-builder.html` 링크 |
| 홈(old) | `/db/` React 앱 iframe 링크 |
| 프로젝트 현황 | 별도 목록 UI 구현 단서 |

### 관리 대상 범위

프론트엔드 정의에서 확인되는 사이트 범위는 TESOL, AI 번역, 통독문서, AI 윤리, 전시/행사, 시험사이트, 전문가매칭, 레벨테스트, 휴텍씨, IITA협회, 반도체/조선/방산 등이다. 대상 업무 역시 문의·견적·번역신청·전문가지원·시험 결과·신청서 템플릿·교육 콘텐츠까지 넓다.

### 추정 업무 흐름

```text
로그인
  -> 실행 홈: 미처리/오늘 접수 파악
  -> 홈페이지 또는 유형 선택
  -> 리스트: 접수 건 상세 확인, 상태 전환, 메모, 편집
  -> 신청서: 접수 폼 및 템플릿 구성
  -> 분석: 분포/성과 확인
  -> 콘텐츠: 문제·커리큘럼·교재 관리
```

## 4. UX 평가

### 강점

| 항목 | 평가 |
| --- | --- |
| 로그인 첫인상 | 불필요한 요소 없이 업무 시스템임을 바로 전달한다. |
| 업무 중심 메뉴 | 실행, 리스트, 신청서, 분석, 콘텐츠가 운영자의 업무 순서와 대체로 맞는다. |
| 데이터 처리 기능 | 검색·필터·상태 변경·메모·상세·페이지네이션 등 관리 도구에 필요한 기능이 폭넓게 정의돼 있다. |
| 다사이트 통합 의도 | 여러 브랜드와 신청 유형을 공통 작업 공간에서 다룰 수 있는 방향은 운영 효율성이 높다. |

### 개선 필요

| 문제 | 영향 | 개선 방향 |
| --- | --- | --- |
| 메뉴와 관리 대상이 매우 많고 `홈(old)`, 샘플/외부 링크가 동일 사이드바에 존재 | 실제 운영 메뉴와 실험 기능의 구분이 어렵고 오작동 위험 | 운영 메뉴, 설정, 실험/도구를 구분하고 권한별로 노출 |
| 56px 아이콘 사이드바와 작은 8~11px 텍스트 중심 UI | 장시간 관리·고해상도·접근성 환경에서 읽기 어렵다 | 펼침형 내비게이션과 본문 최소 글자 크기 상향 |
| 다수 홈페이지·테이블·폼 모듈이 한 화면 모델에 묶임 | 신규 담당자의 학습 부담과 필터 오류 증가 | 사이트군/업무 유형별 작업 공간, 저장 필터, 담당자 큐 제공 |
| 삭제·상태변경·템플릿 수정 기능이 폭넓게 포함 | 권한과 변경 이력이 약하면 운영 사고 가능 | 역할별 권한, 승인, 감사 로그, 복구 가능한 삭제 적용 |

## 5. 보안 및 운영 위험

### 즉시 조치 수준의 관찰

| 우선순위 | 관찰 사실 | 위험 |
| --- | --- | --- |
| Critical | 로그인 카드에 `admin / admin`이 표시됨 | 누구나 관리자 계정으로 로그인 시도 가능 |
| Critical | 공개 JavaScript에 `admin:{pw:"admin", n:"관리자", r:"admin"}` 및 `staff:{pw:"staff", ...}`가 포함됨 | 인증이 브라우저 코드와 `sessionStorage`만으로 이뤄져 우회가 가능하며 비밀번호가 공개됨 |
| Critical | 비로그인 요청 `GET /api/form_templates`가 `200 OK`, JSON 본문 길이 `12122`로 응답 | 최소 템플릿 데이터가 인증 없이 반환되며 다른 API에서도 개인정보/운영 데이터 노출 가능성 |
| High | HTML에 다수의 `POST`, `PUT`, `DELETE` API 호출 코드가 존재 | 서버 측 인가가 없다면 외부 사용자가 데이터 변경·삭제 가능 |
| High | HTTP 제공 및 `Access-Control-Allow-Origin: *` 응답 | 평문 전송과 교차 출처 접근 범위로 위험 증폭 |

### 반드시 먼저 수행할 조치

1. 외부 접근을 즉시 제한하거나 유지보수 페이지로 전환하고 `admin/admin`, `staff/staff` 자격을 폐기한다.
2. 브라우저 단 인증과 `sessionStorage` 기반 역할 판정을 제거하고 서버 세션 또는 검증된 인증 체계를 적용한다.
3. 전체 `/api/*`에 인증·역할별 인가를 추가하고, 특히 신청서/문의/전문가/시험 관련 데이터가 외부에 노출되었는지 로그를 조사한다.
4. 쓰기 API에 CSRF 방어, 입력 검증, 감사 로그, 삭제 복구 정책을 적용한다.
5. HTTPS와 제한적 CORS 정책을 적용한 뒤에만 운영 접근을 허용한다.

## 6. 제품·정보 구조 개선 방향

보안 조치 후에는 시스템의 넓은 범위를 운영자가 이해하기 쉬운 구조로 줄이는 작업이 필요하다.

| 영역 | 현재 단서 | 제안 구조 |
| --- | --- | --- |
| 작업 큐 | 실행 화면에서 미처리·단계·역할 집계 | `내 할 일`, `오늘 접수`, `기한 초과`, `승인 대기` 중심 홈 |
| 사이트 관리 | 다수 홈페이지 칩/테이블 | 브랜드/서비스군 필터와 담당 범위 기본값 |
| 접수 관리 | 리스트와 신청서 메뉴에 기능 분산 | `접수 처리`와 `양식 관리`를 명시적으로 분리 |
| 콘텐츠 | 문제·커리큘럼·교재 CRUD | 교육 콘텐츠 권한이 있는 사용자에게만 독립 메뉴 노출 |
| 실험 기능 | 홈(old), 홈페이지 생성, 외부 AnalyzeAI | 관리자 도구 또는 별도 개발 환경으로 분리 |

## 8. 후속 확인 항목

- `/api/*` 전수 목록과 인증 없는 읽기/쓰기 가능 여부
- API 데이터에 이름, 연락처, 지원서, 시험 결과 등 개인정보가 포함되는지 여부
- 기존 접속 및 변경 로그 보존 여부와 사고 대응 필요성
- 관리자·직원·콘텐츠 담당자 등 역할 정의와 요구 권한
- 운영 대상 사이트 중 실제 사용 중인 사이트와 실험/폐기 대상 분류
